**Omni persona.json** is a portable, composable JSON specification for defining non-human identitiesâ€”AI assistants, brands, bots, mascots, characters, and any other non-human that presents a public-facing identity.

## Relationship to life.json

persona.json complements [life.json](/codex/life-json) as twin specifications for identity duality:

| Spec | Describes | Owned By |
|------|-----------|----------|
| **life.json** | Human identity | The user |
| **persona.json** | Non-human identity | The developer/operator |

They meet in the middle: life.json's `assistants` slice stores the relationship between a specific human and a specific persona (learned facts, permissions, preferences).

## Schema Slices

persona.json uses a modular slice architecture. Only `identity` is requiredâ€”include additional slices as needed:

| Slice | Required | Description |
|-------|----------|-------------|
| `identity` | Yes | Core identity: id, name, tagline, icon, type, description, url |
| `voice` | No | Wake words, TTS/STT provider configuration |
| `personality` | No | System prompt, tone, verbosity, expertise, traits, mode-specific guidelines |
| `branding` | No | Colors (hex), assets (avatar/logo/banner URLs), marketing description |
| `capabilities` | No | Channel-specific tool profiles, permissions, integrations (MCP, APIs) |

## Entity Types

The `identity.type` field classifies the persona:

- **assistant** â€” AI assistants (e.g. Orin)
- **brand** â€” Company or product identities
- **bot** â€” Discord bots, Slack bots, automation
- **character** â€” NPCs, interactive fiction
- **mascot** â€” Brand mascots
- **service** â€” Service-level identity

## Voice Configuration

The `voice` slice configures speech interaction:

- **Wake words** â€” Activation phrases for hands-free interaction
- **TTS** â€” Text-to-speech with provider, voice, speed (0.25â€“4.0x), and pitch (0.5â€“2.0x)
- **STT** â€” Speech-to-text with provider, model, and BCP 47 language code

Supported TTS providers: `openai`, `elevenlabs`, `azure`, `google`, `custom`

Supported STT providers: `openai`, `azure`, `google`, `deepgram`, `custom`

## Capabilities

The `capabilities` slice controls what a persona can do across channels:

**Tool profiles** define available tool access per channel (`voice`, `discord`, `slack`, `telegram`, etc.) with a `default` fallback:

| Profile | Description |
|---------|-------------|
| `none` | No tool access |
| `minimal` | Basic tools only |
| `messaging` | Messaging-appropriate tools |
| `full` | All available tools |
| `custom` | Custom tool configuration |

**Permissions** set global flags: `canLearn`, `canAccessWeb`, `canExecuteCode`, `canAccessFiles`

**Integrations** connect external services like MCP servers and APIs.

## Examples

### Minimal

```json
{
  "$schema": "https://manifold.omni.dev/@omni/schemas/persona.schema.json",
  "version": "1.0.0",
  "identity": {
    "id": "example",
    "name": "Example Entity"
  }
}
```

## Getting Started

1. Create a JSON file with `$schema` pointing to `https://manifold.omni.dev/@omni/schemas/persona.schema.json`
2. Add `version: "1.0.0"` and an `identity` slice with at least `id` and `name`
3. Add optional slices (`voice`, `personality`, `branding`, `capabilities`) as needed
4. Validate against the schema â€” see the [repository](https://github.com/omnidotdev/persona-json) for more examples
