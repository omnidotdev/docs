Vortex is fully open source (Apache 2.0) and designed to run on your own infrastructure. Two deployment options are supported: Docker Compose for simpler setups and Kubernetes (via Helm) for production-grade orchestration.

## Docker Compose

The fastest way to self-host Vortex. The stack repo includes two compose files:

- **`compose.yaml`** pulls pre-built images from `manifold.omni.dev` (Omni's private registry)
- **`compose.dev.yaml`** overrides the image references to build from source instead

For self-hosted deployments, use the dev override to build from the cloned service repos.

### Prerequisites

- [Docker](https://docs.docker.com/get-docker/) with Compose v2
- [Git](https://git-scm.com/)
- A [Hatchet](https://hatchet.run) account or self-hosted instance (workflow executor)

### Quick start

```sh
# Clone the stack repo and service repos
git clone https://github.com/omnidotdev/vortex vortex-stack && cd vortex-stack
git clone https://github.com/omnidotdev/vortex-api services/vortex-api
git clone https://github.com/omnidotdev/vortex-app services/vortex-app
git clone https://github.com/omnidotdev/vortex-worker services/vortex-worker

cp .env.local.template .env.local
# Edit .env.local and fill in required values (see below)

docker compose -f compose.yaml -f compose.dev.yaml up --build
```

The app will be available at `http://localhost:3000`, the API at `http://localhost:4000`.

### Required environment variables

| Variable | Description | Generate with |
|----------|-------------|---------------|
| `DB_PASSWORD` | PostgreSQL password | `openssl rand -hex 16` |
| `CACHE_PASSWORD` | Valkey (Redis-compatible) password | `openssl rand -hex 16` |
| `AUTH_SECRET` | Session signing secret | `openssl rand -hex 16` |
| `ENCRYPTION_KEY` | Data encryption key | `openssl rand -base64 32` |
| `INTERNAL_API_SECRET` | Service-to-service auth | `openssl rand -hex 16` |
| `IGGY_USERNAME` | Event streaming username | Any value |
| `IGGY_PASSWORD` | Event streaming password | Any value |
| `HATCHET_CLIENT_TOKEN` | Hatchet API token | From Hatchet dashboard |

### Optional variables

| Variable | Description | Default |
|----------|-------------|---------|
| `DB_NAME` | Database name | `vortex` |
| `DB_USER` | Database user | `postgres` |
| `DB_PORT` | Database port | `5432` |
| `API_PORT` | API port | `4000` |
| `APP_PORT` | App port | `3000` |
| `CACHE_PORT` | Valkey port | `6379` |
| `EVENTS_TCP_PORT` | Iggy TCP port | `8090` |
| `HATCHET_CLIENT_TLS_STRATEGY` | TLS mode for Hatchet (`none`, `tls`, `mtls`) | `none` |
| `CORS_ALLOWED_ORIGINS` | Allowed CORS origins | `http://localhost:3000` |
| `VORTEX_PUBLIC_URL` | Public URL for webhook callbacks | `http://localhost:4000` |
| `VORTEX_CALLBACK_BASE_URL` | Base URL for OAuth callback redirects | `http://localhost:3000` |
| `PLATFORM_ORG_ID` | Platform org ID for multi-tenant deployments | (empty, single-tenant) |

OAuth login providers (all optional, email/password is always available):

Omni platform integrations (optional, omit to run Vortex standalone):

| Variable | Description |

### Architecture

The Docker Compose stack includes:

- **PostgreSQL 18** for persistent storage
- **Valkey 8** (Redis-compatible) for caching and distributed locks
- **Apache Iggy** for event streaming
- **vortex-api** (port 4000) for the GraphQL and REST API
- **vortex-app** (port 3000) for the web UI
- **vortex-worker** for workflow execution

All services include health checks and proper dependency ordering. The worker depends on the API being healthy before starting.

### Self-hosted features

When billing is not configured, the pricing page shows "All Features" mode, and all entitlements are unlocked. Self-hosted deployments get the full feature set with no artificial limits: unlimited workflows, unlimited executions, all integrations, custom plugins, and SSO.

## Kubernetes (Helm)

For production deployments, Vortex ships a Helm chart with configurable replicas, resource limits, ingress, TLS, and Pod Disruption Budgets.

### Prerequisites

The Helm chart deploys the three Vortex services (API, app, worker) and optionally a PostgreSQL subchart. The following external dependencies must be provisioned separately:

- **Valkey / Redis** for caching and distributed locks. Configure via `externalCache` values, or deploy your own Valkey instance alongside the chart.
- **[Apache Iggy](https://iggy.rs)** for event streaming. The worker and API both require a reachable Iggy server.
- **[Hatchet](https://docs.hatchet.run/self-hosting)** for workflow execution. See the Hatchet self-hosting docs for setup instructions.

### Install

```sh
helm install vortex ./charts/vortex \
  --namespace vortex --create-namespace \
  -f charts/vortex/values.yaml \
  --set secrets.dbPassword=<password> \
  --set secrets.authSecret=<secret> \
  --set secrets.encryptionKey=<key> \
  --set secrets.internalApiSecret=<secret> \
  --set secrets.hatchetClientToken=<token> \
  --set secrets.iggyUsername=<username> \
  --set secrets.iggyPassword=<password> \
  --set secrets.cachePassword=<password>
```

| Secret | Description | Generate with |
|--------|-------------|---------------|
| `secrets.dbPassword` | PostgreSQL password | `openssl rand -hex 16` |
| `secrets.authSecret` | Session signing secret | `openssl rand -hex 16` |
| `secrets.encryptionKey` | Data encryption key | `openssl rand -base64 32` |
| `secrets.internalApiSecret` | Service-to-service auth | `openssl rand -hex 16` |
| `secrets.hatchetClientToken` | Hatchet API token | From Hatchet dashboard |
| `secrets.iggyUsername` | Iggy streaming username | Any value |
| `secrets.iggyPassword` | Iggy streaming password | Any value |
| `secrets.cachePassword` | Valkey/Redis password | `openssl rand -hex 16` |

See `charts/vortex/values.yaml` for the full list of configurable values, and `charts/vortex/values.prod.yaml` for a production reference.

### External database

To use an external PostgreSQL instance instead of the bundled subchart:

```yaml
postgresql:
  enabled: false

externalDatabase:
  host: your-db-host
  port: 5432
  database: vortex
  username: postgres
  password: <password>
```

## Upgrading

Pull the latest changes and rebuild:

```sh
git pull
docker compose -f compose.yaml -f compose.dev.yaml up --build
```

Database migrations run automatically on startup.

## Troubleshooting

**Services won't start**: Check that all required env vars in `.env.local` are set. The compose file will print which variable is missing (e.g., `DB_PASSWORD is required`).

**Database connection errors**: Verify `DB_PASSWORD` matches between the `db` service and `api`/`worker` services (they derive the URL from the same env var).

**Hatchet connection failures**: Ensure `HATCHET_CLIENT_TOKEN` is valid and the Hatchet server is reachable. Set `HATCHET_CLIENT_TLS_STRATEGY=none` for local Hatchet instances without TLS.

**Worker not starting**: The worker depends on the API being healthy first. Check the API logs for errors before investigating the worker.
