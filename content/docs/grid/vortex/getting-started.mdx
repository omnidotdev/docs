Get up and running with Vortex in a few minutes. Choose between Docker Compose for local development or Helm for Kubernetes deployments.

## Prerequisites

- [Docker](https://docs.docker.com/get-docker/) and Docker Compose v2
- [Git](https://git-scm.com/)
- A [Hatchet](https://docs.hatchet.run/self-hosting) instance (local or cloud) for workflow execution

## 1. Start the stack

```bash
# Clone the stack repo (includes compose files)
git clone https://github.com/omnidotdev/vortex vortex-stack
cd vortex-stack

# Clone each service into the services/ directory
git clone https://github.com/omnidotdev/vortex-api services/vortex-api
git clone https://github.com/omnidotdev/vortex-app services/vortex-app
git clone https://github.com/omnidotdev/vortex-worker services/vortex-worker

# Copy the environment template and fill in required secrets
cp .env.local.template .env.local
# Edit .env.local: generate secrets with the commands shown in the
# template comments, and set HATCHET_CLIENT_TOKEN from your Hatchet
# instance.

# Build from source and start all services
docker compose -f compose.yaml -f compose.dev.yaml up --build
```

This starts PostgreSQL 18, Valkey 8 (Redis-compatible cache), Apache Iggy (event streaming), and the three Vortex services (API, app, worker). Once healthy, the dashboard is available at `http://localhost:3000` and the API at `http://localhost:4000`.

See the [Self-Hosting guide](/grid/vortex/self-hosting) for Kubernetes and production deployment options.

## 2. Create a workflow

Vortex supports three authoring modes. Every format round-trips to the others.

### Visual editor

Open the dashboard at `http://localhost:3000`, navigate to **Workflows**, and click **New Workflow**. Drag steps from the sidebar, connect them with edges, and configure each step inline.

### JSON DSL

Paste or import a JSON definition into the editor's import dialog:

```json
{
  "version": "1.0",
  "steps": [
    {
      "id": "trigger_1",
      "type": "trigger",
      "name": "Webhook",
      "trigger": { "type": "webhook", "config": {} }
    },
    {
      "id": "notify",
      "type": "action",
      "name": "Send Slack message",
      "action": {
        "integrationId": "slack",
        "operation": "sendMessage",
        "inputs": {
          "channel": "#alerts",
          "text": "New event: {{triggerData.body.event}}"
        }
      }
    }
  ],
  "edges": [
    { "id": "e-trigger-notify", "source": "trigger_1", "target": "notify" }
  ]
}
```

### TypeScript SDK

Install the SDK and define workflows programmatically:

```bash
bun add @omnidotdev/vortex-sdk
```

```typescript
import { workflow, webhookTrigger } from "@omnidotdev/vortex-sdk";

export default workflow
  .name("Order Processor")
  .trigger(webhookTrigger({ path: "/orders" }))
  .step("action", "Validate Order")
    .config({ integration: "validator", operation: "check", inputs: { id: "$.orderId" } })
  .end()
  .step("action", "Charge Payment")
    .config({ integration: "stripe", operation: "charge", inputs: { amount: "$.total" } })
  .build();
```

## 3. Trigger a run

### Webhook

Every workflow with a webhook trigger gets a unique URL:

```bash
curl -X POST http://localhost:4000/webhooks/workflow/<workflowId>/<secret> \
  -H "Content-Type: application/json" \
  -d '{"event": "order.created", "orderId": "ord_123"}'
```

### Cron

Set a cron expression on the trigger and Vortex fires the workflow automatically. The scheduler uses distributed locking for multi-instance safety.

### Event

Subscribe to events from any Omni service. The event router matches incoming events by type pattern, optional conditions, and optional transforms.

### Manual

Trigger a run from the UI or via API:

```bash
curl -X POST http://localhost:4000/api/v1/workflows/<workflowId>/trigger \
  -H "Authorization: Bearer <api-key>" \
  -H "Content-Type: application/json" \
  -d '{"data": {"userId": "u_123"}}'
```

## 4. Monitor execution

The dashboard shows real-time workflow run status, step-by-step execution traces, and variable state. Runs also emit lifecycle events (`vortex.workflow.started`, `vortex.workflow.completed`, `vortex.workflow.failed`) that can trigger other workflows.
