Synapse routes requests to the best provider and model using configurable strategies. This page covers routing configuration, strategies, and virtual model names.

## Routing Configuration

Enable routing and choose a strategy in the `[llm.routing]` section. Register available models with cost and quality metadata:

```toml
[llm.routing]
enabled = true
strategy = "threshold"

[llm.routing.threshold]
low_complexity_model = "openai/gpt-4o-mini"
high_complexity_model = "anthropic/claude-sonnet-4-20250514"
quality_floor = 0.7

[[llm.routing.models]]
provider = "openai"
model = "gpt-4o-mini"
input_per_mtok = 0.15
output_per_mtok = 0.60
quality = 0.7
context_window = 128000

[llm.routing.models.capabilities]
tool_calling = true
vision = true

[[llm.routing.models]]
provider = "anthropic"
model = "claude-sonnet-4-20250514"
input_per_mtok = 3.00
output_per_mtok = 15.00
quality = 0.92
context_window = 200000

[llm.routing.models.capabilities]
tool_calling = true
vision = true
long_context = true
```

### Model Fields

| Field | Description |
|-------|-------------|
| `provider` | Provider name matching a configured `[llm.providers.*]` entry |
| `model` | Model identifier as the provider expects it |
| `input_per_mtok` | Cost per million input tokens (USD) |
| `output_per_mtok` | Cost per million output tokens (USD) |
| `quality` | Quality score from 0.0 to 1.0 |
| `context_window` | Maximum context length in tokens |
| `capabilities` | Optional capability flags (`tool_calling`, `vision`, `long_context`) |

## Routing Strategies

### Threshold (default)

Routes low-complexity requests to a fast/cheap model and high-complexity requests to a frontier model, based on a quality floor:

```toml
[llm.routing]
enabled = true
strategy = "threshold"

[llm.routing.threshold]
low_complexity_model = "openai/gpt-4o-mini"
high_complexity_model = "anthropic/claude-sonnet-4-20250514"
quality_floor = 0.7
```

### Cost

Always select the cheapest model that meets minimum capability requirements:

```toml
[llm.routing]
enabled = true
strategy = "cost"
```

Synapse compares `input_per_mtok` and `output_per_mtok` across all registered models and picks the lowest combined cost.

### Cascade (Buffered Streaming)

Try the fastest/cheapest model first. If the response confidence is below a threshold, discard it and escalate to a stronger model. Supports buffered streaming so the client sees a seamless stream:

```toml
[llm.routing]
enabled = true
strategy = "cascade"

[llm.routing.cascade]
initial_model = "openai/gpt-4o-mini"
escalation_model = "anthropic/claude-sonnet-4-20250514"
confidence_threshold = 0.5
max_buffer_bytes = 32768
buffer_timeout_secs = 10
```

| Field | Description | Default |
|-------|-------------|---------|
| `initial_model` | Cheap model to try first (`"provider/model"`) | Cheapest registered model |
| `escalation_model` | Stronger model to escalate to (`"provider/model"`) | Best quality registered model |
| `confidence_threshold` | Confidence score below which to escalate (0.0 to 1.0) | 0.5 |
| `max_buffer_bytes` | Maximum bytes to buffer before committing | 32768 (32 KB) |
| `buffer_timeout_secs` | Seconds to wait for buffered response before committing | 10 |

<Callout type="info">
  Cascade strategy buffers the initial response before streaming to the client. This adds slight latency but allows Synapse to transparently escalate to a better model if the initial response shows low confidence (hedging language, insufficient detail).
</Callout>

### Score (Multi-Objective)

Compute a weighted score across quality, cost, and latency dimensions and select the highest-scoring model:

```toml
[llm.routing]
enabled = true
strategy = "score"

[llm.routing.score]
weight_quality = 0.5
weight_cost = 0.3
weight_latency = 0.2
error_penalty = 1.0
min_samples = 10
```

| Field | Description | Default |
|-------|-------------|---------|
| `weight_quality` | Weight for quality component (0.0 to 1.0) | 0.5 |
| `weight_cost` | Weight for cost component (0.0 to 1.0) | 0.3 |
| `weight_latency` | Weight for latency component (0.0 to 1.0) | 0.2 |
| `error_penalty` | Error rate penalty multiplier | 1.0 |
| `min_samples` | Minimum request samples before feedback influences scoring | 10 |

The final score for each model is `weight_quality * quality + weight_cost * (1 - normalized_cost) + weight_latency * (1 - normalized_latency)`, with an optional error rate penalty applied from runtime feedback.

### ONNX (ML-Based)

Use a custom ONNX model to classify query complexity and select the optimal model. The ONNX strategy extracts features from each query (token count, task type, complexity, capabilities, message count) and runs inference via the `ort` runtime.

ONNX routing requires building Synapse with the `onnx` feature flag:

```bash
cargo build --release --features onnx
```

Then load a pre-trained model at startup:

```rust
// ONNX strategy is loaded programmatically, not via TOML config
let strategy = OnnxStrategy::load("/etc/synapse/routing_model.onnx")?;
```

<Callout type="warning">
  ONNX routing requires the `onnx` compile-time feature flag. Without it, the strategy returns a feature-not-available error. You need to train and export your own routing model offline in Python; see the repository for example training scripts.
</Callout>

## Virtual Model Names

Synapse provides built-in virtual model names that map to routing strategies:

| Virtual Model | Description |
|---------------|-------------|
| `auto` | Default routing, uses threshold strategy |
| `fast` | Optimizes for lowest latency |
| `best` | Optimizes for highest quality |
| `cheap` | Optimizes for lowest cost |

Clients can use these names directly:

```bash
curl http://localhost:6000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "auto",
    "messages": [{"role": "user", "content": "Hello!"}]
  }'
```

## Passthrough Models

If a request specifies a model name that doesn't match any virtual name, Synapse passes it through to the appropriate provider based on model name prefix matching (e.g. `claude-*` routes to Anthropic, `gpt-*` routes to OpenAI) or explicit `provider/model` format (e.g. `anthropic/claude-sonnet-4-20250514`).

See also: [Providers](/grid/synapse/providers) for configuring the upstream providers that routing selects from.
